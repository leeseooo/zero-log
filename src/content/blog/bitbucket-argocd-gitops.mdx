Bitbucket Pipelines → ArgoCD(GitOps)로 운영 서버에 배포되는 흐름을 뜯어봤어요. 처음엔 파이프라인이 서버에 직접 뭔가 꽂는 줄 알았는데, 실제론 전혀 아니더라고요.

## 배포 핵심 흐름 (4 스텝)

```yaml
- step:
    name: Deploy to PRODUCTION
    image:
      name: <account-id>.dkr.ecr.ap-northeast-2.amazonaws.com/.../bitbucket-pipelines-nextjs:latest
      aws:
        oidc-role: $MGT_ECR_ROLE_ARN
    oidc: true
    deployment: production
    trigger: manual
    services: [docker]
    clone:
      enabled: false
    script:
      - export APP_NAME=...
      - export NAMESPACE=web
      - source /scripts/step-deploy-application.sh
```

1. **배포 전용 이미지 로드** — AWS CLI, kubectl, kustomize 같은 도구가 미리 깔린 Docker 이미지를 환경으로 사용
2. **안전한 인증(OIDC)** — 비밀번호를 노출하지 않고 `OIDC Role` 로 AWS 권한을 일회성으로 획득
3. **배포 스크립트 실행** — 배포 대상 정보를 환경 변수로 세팅하고, 스크립트가 **GitOps 레포의 이미지 태그 정보**를 업데이트
4. **ArgoCD 동기화** — ArgoCD가 Git 변경을 감지하고, 실제 운영 서버(K8s) 상태를 Git과 동일하게 맞춤(Sync)

## 주요 설정값의 의미

- `trigger: manual` — 운영 배포 안정성을 위해 관리자가 직접 승인(버튼 클릭)할 때만 진행
- `clone: enabled: false` — 소스를 빌드하는 단계가 아니라 **설정만 바꾸는 단계**라, 무거운 소스 복제를 생략해 속도를 높임
- `after-script` — 배포 성공/실패를 판단해 알림·후속 처리

## 깨달음

가장 크게 와닿은 지점이에요.

> 배포 파이프라인이 서버에 파일을 직접 복사하는 게 아니라, **ArgoCD가 바라보는 '설정 장부(Git)'를 업데이트**함으로써 간접적으로 배포가 완성된다.

Bitbucket Pipeline은 "장부에 새 이미지 태그를 적는" 역할만 하고, 실제 서버 상태를 Git과 일치시키는 건 ArgoCD 몫이에요. `clone: enabled: false` 가 왜 있는지도 이걸 알고 나니 납득됐어요 — 소스를 만질 일이 없으니까요. "배포 = 서버에 직접 뭔가 한다"는 감각을 "배포 = 원하는 상태를 Git에 선언한다"로 바꾸니 전체 그림이 열렸습니다.
