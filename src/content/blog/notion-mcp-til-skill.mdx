## 문제

Claude Code랑 대화하다 새로 알게 된 걸 매번 손으로 노션에 옮겨 적는 게 은근 번거로웠어요. **노션 링크만 던지면 거기다 TIL을 자동으로 써주는 스킬**이 있으면 좋겠다 싶었습니다.

## 해결

`write-a-skill` 가이드를 따라 `~/.claude/skills/til-notion/SKILL.md` 를 작성했어요. 핵심은 노션 DB에 "행 추가"하는 흐름이었는데, 이게 세 단계였어요.

1. `notion-fetch` 로 대상 페이지를 조회 → 인라인 DB의 `data-source-url`(`collection://...`)을 추출
2. 그 data source를 다시 fetch → 스키마(title 속성명 등)를 확인. **DB마다 컬럼명이 달라서 추측하면 안 돼요.**
3. `notion-create-pages` 에 `parent: { type: "data_source_id", data_source_id }` 로 페이지 1개를 생성 = DB의 새 행

기본 TIL Box DB는 스킬에 박아두되, 다른 링크를 주면 그 링크를 우선하도록 분기했어요.

## 교훈

만들면서 400 에러로 몇 번 튕기며 배운 것들이에요.

- 노션에서 **"DB에 행 추가" = 그 data source를 부모로 하는 페이지 생성**이에요. `database_id` 가 아니라 `data_source_id` 를 써야 합니다. 이거 헷갈려서 한참 헤맸어요.
- `created_time` 같은 자동/동기 속성은 값을 넣으면 안 돼요 (readOnly).
- **multi_select(태그) 속성은 미리 정의된 옵션만 넣을 수 있어요.** 옵션 목록이 비어 있으면 새 값을 그대로 못 넣고 400 validation 에러가 납니다. 먼저 data source에 옵션을 추가하거나, 태그를 아예 생략해야 해요.
- 날짜·우선순위 같은 동적 정보는 스킬 문서에 하드코딩하지 않고 런타임에 확인하게 했어요. (정적인 규칙과 동적인 데이터를 나누는 감각 — 이른바 Software 3.0 방식)

## 남길 판단

스킬을 만들다 보면 결국 "이 정보는 문서에 고정할 것 vs 실행 때 확인할 것"을 나누는 게 핵심이더라고요. DB 스키마나 오늘 날짜처럼 바뀌는 건 런타임에 물어보게 하고, 흐름과 규칙만 문서에 남기니까 다른 노션 DB에도 그대로 재사용됐어요.

참고로 이 블로그도 노션 트러블슈팅 글을 옮겨와 만들고 있는데, 이런 자동화가 쌓이니까 "기록 → 정리 → 공개"까지의 마찰이 확 줄어요.
