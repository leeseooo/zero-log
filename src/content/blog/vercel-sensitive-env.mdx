Vercel 보안 사고 소식을 계기로 환경 변수 저장 방식을 다시 들여다봤어요. 두 종류를 확실히 구분해야 하더라고요.

## 1. Non-sensitive (민감하지 않은 변수)

노출돼도 서비스에 치명타가 적은 설정값이에요. 보통 **plaintext로 저장**돼요.

- **특징:** Vercel 대시보드 환경 변수 목록에서 **실제 값을 바로 확인**할 수 있어요.
- **예시:** `NEXT_PUBLIC_API_URL`, `THEME_COLOR`, `GA_TRACKING_ID`
- **사고 관련:** 이번 사고에서 공격자가 가져간 게 바로 이 유형이에요. "이건 별로 안 중요해 보여"라고 방치한 값들이, 공격자에겐 **시스템 구조를 파악하는 지도**가 된 셈이죠.

## 2. Sensitive (민감한 변수 / Secrets)

보안이 필수인 핵심 정보예요. 저장될 때부터 **암호화**되고 시스템 내부에서도 엄격하게 관리돼요.

- **특징:** 등록하는 순간 값이 `••••••` 로 가려져요. **한 번 저장하면 관리자라도 원본 값을 다시 못 봐요.** 오직 빌드 프로세스 중에만 메모리에 로드됩니다.
- **예시:** `AWS_SECRET_ACCESS_KEY`, `DB_PASSWORD`, `STRIPE_API_KEY`
- **보안성:** 이번 사고에서도 이 영역 값들은 암호화 덕분에 공격자가 쉽게 가져가기 어려웠을 거라고 평가받아요.

## 한눈에 비교

| 구분 | Non-sensitive (Plaintext) | Sensitive (Secret) |
|---|---|---|
| 가독성 | 대시보드에서 누구나 확인 | `••••` 처리, 확인 불가 |
| 주요 용도 | 단순 설정값, 공개 API 주소 | 인증 토큰, 비밀번호, 암호키 |
| 편집 | 값을 바로 수정·확인 | 지우고 새로 입력만 가능 |
| 사고 시 위험 | 즉시 탈취·이용 가능 | 암호화돼 탈취해도 해독 어려움 |

## 남길 판단

핵심은 이거예요. **"공개돼도 되는 값"과 "절대 안 되는 값"을 등록 시점에 구분**하는 것. 토큰·비밀번호·암호키는 반드시 Sensitive로 넣고, plaintext 변수엔 시스템 구조를 유추할 단서를 최대한 안 남기는 게 안전해요. 별거 아니라고 방치한 `NEXT_PUBLIC_` 값 하나가 지도가 될 수 있으니까요.
