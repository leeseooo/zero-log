> 사내 식별자(도메인·패키지명 등)는 공개용으로 일반화했어요. 원칙 위주로 봐주세요.

AI 에이전트(Codex 등)한테 우리 모노레포에서 일을 시키다 보니, 매번 같은 걸 설명하고 있더라고요. 그래서 에이전트 전용 지침서 `AGENTS.md` 를 만들었어요. 만들면서 든 생각은 하나였어요 — **규칙을 빠짐없이 적는 게 아니라, "언제 무엇을 읽고 어떻게 검증할지"를 설계하는 것**이 핵심이라는 거요.

## 1. 큰 요청은 작게 쪼개게

한 방에 끝내려 하지 말고 `설계 → 최소 구현 → 테스트 추가 → 리팩터링` 처럼 작은 목표로 나누게 했어요. 그리고 작업 시작 때 **입력 컨텍스트**(봐야 할 파일·로그·테스트)와 **완료 조건**(통과해야 할 테스트/린트/타입체크 명령)을 함께 못박게 했습니다.

## 2. "완료"는 말로 검증하지 않는다

이게 제일 강조한 부분이에요.

> 완료는 말로 검증하지 않습니다. 가능한 경우 실제 명령 실행 결과로 확인합니다.

에이전트가 "다 됐습니다" 하고 끝내는 걸 막고, `관련 파일 찾기 → 변경 방향 정리 → 수정 → 검증 명령 실행 → 결과 요약` 루프를 기본값으로 뒀어요. 모르는 건 추측하지 말고 파일·테스트·로그·명령 출력을 먼저 확인하게 하고요.

## 3. 문서는 lazy-load — 다 읽지 마

모노레포엔 패키지마다 `CLAUDE.md` 가 있는데, 이걸 다 읽으면 토큰만 태워요. 그래서 규칙을 뒀어요.

- 수정 대상 경로의 **가장 가까운** 패키지 문서만 필요할 때 읽기
- 전역 판단이 필요할 때만 루트 규칙 문서 확인
- 단순 탐색·질문·한 줄 수정에선 아무 문서도 안 읽어도 됨

코드 탐색도 `rg` 나 대량 `Read` 보다 CodeGraph(심볼 정의·호출 관계·영향 범위 질의)를 먼저 쓰고, 후보를 좁힌 뒤 필요한 파일만 부분적으로 읽게 했어요.

## 4. 보고는 diff 중심으로

최종 보고 형식을 고정했어요 — **변경 파일 목록 / 핵심 변경 요약 / 실행한 검증 명령과 결과 / 리스크·사이드이펙트 / (필요시) 롤백 방법.** 단순 수정은 짧게 합쳐도 되지만, **리팩터링·권한·결제·인프라·보안 변경은 리스크와 롤백을 절대 생략 못 하게** 했어요.

## 5. 사람을 부르는 기준(HITL)

에이전트가 알아서 다 하면 안 되는 것들이 있잖아요. 이건 반드시 사용자 확인을 요청하게 했어요.

- 보안 관련 코드 변경
- API 계약 변경
- 10개 이상 파일 변경
- 100줄 이상 코드 삭제
- DB 스키마 변경

## 남길 판단

`AGENTS.md` 를 쓰다 보니, 결국 [노션 TIL 스킬을 만들 때](/blog/notion-mcp-til-skill) 배운 거랑 똑같은 결론에 도달했어요. **정적인 규칙(코드 스타일, FSD 레이어 방향, 검증 명령)은 문서에 고정하고, 동적인 판단(어떤 문서를 읽을지, 무엇을 검증할지)은 런타임에 하게 나누는 것.** 규칙을 많이 적을수록 좋은 게 아니라, 에이전트가 "지금 이 작업에 필요한 최소한"만 집어들게 설계하는 게 좋은 AGENTS.md였어요.
